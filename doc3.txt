Mockito Hands-On Exercises
Exercise 1: Mocking and Stubbing
Scenario:
You need to test a service that depends on an external API. Use Mockito to mock the external API and stub its methods.
Steps:
Create a mock object for the external API.
Stub the methods to return predefined values.
Write a test case that uses the mock object.
Solution Code:
import static org.mockito.Mockito.*; import org.junit.jupiter.api.Test; import org.mockito.Mockito;
public class MyServiceTest { @Test
public void testExternalApi() {
ExternalApi mockApi = Mockito.mock(ExternalApi.class); when(mockApi.getData()).thenReturn("Mock Data"); MyService service = new MyService(mockApi);
String result = service.fetchData(); assertEquals("Mock Data", result);
}
}

java
package com.example;
import static org.junit.jupiter.api.Assertions.assertEquals;
import static org.mockito.Mockito.*;
import org.junit.jupiter.api.BeforeEach;
import org.junit.jupiter.api.Test;
public class MyServiceTest {
    
    private ExternalApi mockApi;
    private MyService service;
    
    @BeforeEach
    void setUp() {
        // 1. CREATE MOCK: Create a mock object for the external API
        mockApi = mock(ExternalApi.class);
        
        // Create the service with the mock
        service = new MyService(mockApi);
    }
    
    @Test
    public void testFetchData() {
        // 2. STUB: Define what the mock should return when called
        when(mockApi.getData()).thenReturn("Mock Data");
        
        // 3. EXECUTE: Call the service method
        String result = service.fetchData();
        
        // 4. VERIFY: Check the result
        assertEquals("Mock Data", result);
        
        // Optional: Verify the method was called
        verify(mockApi).getData();
    }
    
    @Test
    public void testFetchDataMultipleStubs() {
        // Stub with different return values
        when(mockApi.getData())
            .thenReturn("First Call")
            .thenReturn("Second Call");
        
        assertEquals("First Call", service.fetchData());
        assertEquals("Second Call", service.fetchData());
    }
    
    @Test
    public void testCheckStatus() {
        // Stub another method
        when(mockApi.getStatusCode()).thenReturn(200);
        
        int status = service.checkStatus();
        
        assertEquals(200, status);
        verify(mockApi).getStatusCode();
    }
}

import static org.mockito.Mockito.*;
This imports all static Mockito methods like mock(), when(), verify(), etc.
java
import org.junit.jupiter.api.Test;
Imports the @Test annotation from JUnit 5.
java
public class MyServiceTest {
    @Test
    public void testExternalApi() {
        // 1. CREATE MOCK
        ExternalApi mockApi = Mockito.mock(ExternalApi.class);
        // Creates a fake implementation of ExternalApi
        // All methods return null/0/false by default
        
        // 2. STUB METHOD
        when(mockApi.getData()).thenReturn("Mock Data");
        // "When getData() is called, return 'Mock Data'"
        // This overrides the default null return
        
        // 3. INJECT MOCK
        MyService service = new MyService(mockApi);
        // Service uses the mock, not the real implementation
        
        // 4. EXECUTE
        String result = service.fetchData();
        // fetchData() calls mockApi.getData()
        // Returns "Mock Data" (not null!)
        
        // 5. ASSERT
        assertEquals("Mock Data", result);
        // Verify the result is what we expected
    }
}

Exercise 2: Verifying Interactions

Steps:

Verify the interaction.
Solution Code:

}

📝 Step 2: Create the Test Class with Verification
Create a new test class or add to existing MyServiceTest.java:
java
package com.example;
import static org.junit.jupiter.api.Assertions.*;
import static org.mockito.Mockito.*;
import org.junit.jupiter.api.BeforeEach;
import org.junit.jupiter.api.Test;
public class MyServiceTest {
    
    private ExternalApi mockApi;
    private MyService service;
    
    @BeforeEach
    void setUp() {
        // Create the mock
        mockApi = mock(ExternalApi.class);
        service = new MyService(mockApi);
    }
    
    // ============================================
    // BASIC VERIFICATION
    // ============================================
    
    @Test
    public void testVerifyInteraction() {
        // 1. Call the method
        service.fetchData();
        
        // 2. Verify the interaction
        verify(mockApi).getData();
        // This passes because getData() was called exactly once
    }
    
    // ============================================
    // VERIFY WITH SPECIFIC ARGUMENTS
    // ============================================
    
    @Test
    public void testVerifyWithArguments() {
        // Call with specific arguments
        service.registerUser("john_doe", "password123");
        
        // Verify the method was called with EXACT arguments
        verify(mockApi).saveUser("john_doe", "password123");
        // This will fail if arguments don't match exactly!
    }
    
    // ============================================
    // VERIFY WITH ARGUMENT MATCHERS
    // ============================================
    
    @Test
    public void testVerifyWithMatchers() {
        service.registerUser("alice", "secret456");
        
        // Verify with matchers (more flexible)
        verify(mockApi).saveUser(anyString(), anyString());
        // anyString() matches ANY string value
        
        // More specific matchers
        verify(mockApi).saveUser(startsWith("alice"), contains("456"));
        // Matches: first arg starts with "alice", second contains "456"
    }
    
    // ============================================
    // VERIFY CALL COUNT
    // ============================================
    
    @Test
    public void testVerifyCallCount() {
        // Call the method multiple times
        service.fetchData();
        service.fetchData();
        service.fetchData();
        
        // Verify called exactly 3 times
        verify(mockApi, times(3)).getData();
        
        // Other count verifications:
        // verify(mockApi, times(1)).getData();  // exactly once
        // verify(mockApi, atLeast(2)).getData(); // 2 or more times
        // verify(mockApi, atMost(4)).getData();  // 4 or fewer times
        // verify(mockApi, never()).getData();    // 0 times (never called)
    }
    
    // ============================================
    // VERIFY WITH DIFFERENT ARGUMENTS
    // ============================================
    
    @Test
    public void testVerifyMultipleCallsWithDifferentArgs() {
        // Call with different arguments
        service.registerUser("user1", "pass1");
        service.registerUser("user2", "pass2");
        
        // Verify first call
        verify(mockApi).saveUser("user1", "pass1");
        
        // Verify second call
        verify(mockApi).saveUser("user2", "pass2");
    }
    
    // ============================================
    // VERIFY NO INTERACTIONS
    // ============================================
    
    @Test
    public void testVerifyNoInteractions() {
        // Don't call any method
        
        // Verify NO interactions with the mock
        verifyNoInteractions(mockApi);
        // This passes because we never called any method
        
        // Or verify no more interactions after specific ones
        service.fetchData();
        verify(mockApi).getData();
        verifyNoMoreInteractions(mockApi);
        // This verifies ONLY getData() was called and nothing else
    }
    
    // ============================================
    // VERIFY ORDER OF CALLS
    // ============================================
    
    @Test
    public void testVerifyOrder() {
        // Call methods in a specific order
        service.fetchData();
        service.checkStatus();
        service.saveData("test");
        
        // Create an in-order verifier
        InOrder inOrder = inOrder(mockApi);
        
        // Verify calls happened in this exact order
        inOrder.verify(mockApi).getData();
        inOrder.verify(mockApi).getStatusCode();
        inOrder.verify(mockApi).updateData("test");
        
        // If the order was different, this test would fail
    }
}

🎯 All Verification Methods Explained
1. Basic Verification
java
verify(mockApi).getData();
Checks method was called exactly once
Fails if called 0 times or 2+ times
2. Times Verification
java
verify(mockApi, times(3)).getData();     // Called exactly 3 times
verify(mockApi, atLeast(2)).getData();   // Called 2 or more
verify(mockApi, atMost(5)).getData();    // Called 5 or fewer
verify(mockApi, never()).getData();      // Called 0 times
verify(mockApi, only()).getData();       // Called only once, no other calls
3. Argument Matchers
java
// Basic matchers
anyString()    // Any String value
anyInt()       // Any int
anyLong()      // Any long
any()          // Any object
eq("exact")    // Exact value (use with matchers)
// String matchers
contains("text")    // Contains substring
startsWith("prefix")
endsWith("suffix")
matches("regex")
// Collection matchers
anyList()
anySet()
anyMap()
anyCollection()
// Custom matcher
argThat(argument -> argument.length() > 5)
4. Combined Examples
java
@Test
public void testCombinedVerification() {
    service.fetchData();      // Calls getData()
    service.checkStatus();     // Calls getStatusCode()
    service.fetchData();      // Calls getData() again
    service.fetchData();      // Calls getData() again
    
    // Verify specific count
    verify(mockApi, times(3)).getData();
    verify(mockApi, times(1)).getStatusCode();
    
    // Verify with matchers
    verify(mockApi, atLeast(2)).getData();
    
    // Verify order
    InOrder inOrder = inOrder(mockApi);
    inOrder.verify(mockApi).getData();
    inOrder.verify(mockApi).getStatusCode();
    inOrder.verify(mockApi, times(2)).getData();
}
⚠️ Common Verification Errors
Error 1: Wrong Arguments
java
// ❌ This will fail
service.registerUser("john", "pass");
verify(mockApi).saveUser("jane", "pass");
// Error: Argument mismatch!
Error 2: Wrong Call Count
java
// ❌ This will fail
service.fetchData(); // only 1 call
verify(mockApi, times(2)).getData();
// Error: Wanted 2 times, but was 1 time
Error 3: Unexpected Calls
java
// ❌ This will fail
service.fetchData();
service.checkStatus(); // Extra call
verify(mockApi).getData();
verifyNoMoreInteractions(mockApi);
// Error: Unwanted interaction: getStatusCode()

📝 Step 2: Test Class with Argument Matchers
java
package com.example;
import static org.junit.jupiter.api.Assertions.*;
import static org.mockito.Mockito.*;
import static org.mockito.ArgumentMatchers.*;
import java.util.*;
import org.junit.jupiter.api.BeforeEach;
import org.junit.jupiter.api.Test;
public class MyServiceTest {
    
    private ExternalApi mockApi;
    private MyService service;
    
    @BeforeEach
    void setUp() {
        mockApi = mock(ExternalApi.class);
        service = new MyService(mockApi);
    }
    
    // ============================================
    // BASIC ARGUMENT MATCHERS
    // ============================================
    
    @Test
    public void testBasicArgumentMatchers() {
        // Call the method
        service.registerUser("john_doe", "password123");
        
        // Verify with ANY matchers
        verify(mockApi).saveUser(anyString(), anyString());
        // anyString() - matches any String value
        
        // Verify with specific types
        service.modifyUser(1, "John", "john@email.com");
        verify(mockApi).updateUser(anyInt(), anyString(), anyString());
    }
    
    // ============================================
    // SPECIFIC STRING MATCHERS
    // ============================================
    
    @Test
    public void testStringMatchers() {
        service.registerUser("alice_wonder", "secret123");
        
        // startsWith - matches strings that start with prefix
        verify(mockApi).saveUser(startsWith("alice"), anyString());
        
        // endsWith - matches strings that end with suffix
        verify(mockApi).saveUser(anyString(), endsWith("123"));
        
        // contains - matches strings that contain substring
        verify(mockApi).saveUser(contains("_wonder"), anyString());
        
        // matches - regex pattern matching
        verify(mockApi).saveUser(matches("^[a-z]+_[a-z]+$"), anyString());
        // ^[a-z]+_[a-z]+$  matches: lowercase_underscore_lowercase
    }
    
    // ============================================
    // NUMBER MATCHERS
    // ============================================
    
    @Test
    public void testNumberMatchers() {
        service.modifyUser(25, "Jane", "jane@email.com");
        
        // anyInt() - matches any int
        verify(mockApi).updateUser(anyInt(), anyString(), anyString());
        
        // eq() - exact match when using matchers
        verify(mockApi).updateUser(anyInt(), eq("Jane"), eq("jane@email.com"));
        
        // gt() - greater than
        service.searchPerson("John", "Doe", 30);
        verify(mockApi).findUser(anyString(), anyString(), gt(18));  // age > 18
        
        // lt() - less than
        service.searchPerson("Mary", "Smith", 15);
        verify(mockApi).findUser(anyString(), anyString(), lt(18));  // age < 18

⚠️ Common Pitfalls and Solutions
Pitfall 1: Mixing Matchers and Exact Values
java
// ❌ WRONG - Can't mix matchers and exact values
verify(mockApi).saveUser("john", anyString());  // ERROR!
// ✅ CORRECT - Use eq() for exact values
verify(mockApi).saveUser(eq("john"), anyString());
// OR use all matchers
verify(mockApi).saveUser(anyString(), anyString());
Pitfall 2: Incorrect Type in Matchers
java
// ❌ WRONG - Type mismatch
verify(mockApi).updateUser(anyInt(), anyString(), anyInt()); // ERROR!
// ✅ CORRECT - Use correct types
verify(mockApi).updateUser(anyInt(), anyString(), anyString());
Pitfall 3: Null Values
java
// ❌ WRONG - anyString() doesn't match null
verify(mockApi).saveUser(null, anyString()); // Won't match!
// ✅ CORRECT - Use isNull() for null values
verify(mockApi).saveUser(isNull(), anyString());
// Or nullable() for null or String
verify(mockApi).saveUser(nullable(String.class), anyString());

Exercise 4: Handling Void Methods
Scenario:
You need to test a void method that performs some action.
Steps:
Create a mock object.
Stub the void method.
Verify the interaction.
Why Are Void Methods Special?
Unlike methods that return values, void methods:
Don't return anything (obviously!)
Usually perform side effects (database updates, logging, API calls)
Can't use when().thenReturn() directly
Need special syntax for stubbing and verification

package com.example;
import java.util.List;
import java.util.Map;
public class MyService {
    private ExternalApi externalApi;
    
    public MyService(ExternalApi externalApi) {
        this.externalApi = externalApi;
    }
    
    // ============================================
    // METHODS THAT CALL VOID METHODS
    // ============================================
    
    public void saveData(String data) {
        externalApi.updateData(data);
    }
    
    public void registerUser(String username, String password) {
        if (username == null || username.trim().isEmpty()) {
            throw new IllegalArgumentException("Username cannot be empty");
        }
        externalApi.saveUser(username, password);
        externalApi.logEvent("USER_REGISTERED", Map.of("username", username));
    }
    
    public void deleteUser(long userId) {
        if (userId <= 0) {
            throw new IllegalArgumentException("Invalid user ID");
        }
        externalApi.deleteRecord(userId);
        externalApi.logEvent("USER_DELETED", Map.of("userId", userId));
    }
    
    public void notifyUser(String email, String subject, String message) {
        externalApi.sendEmail(email, subject, message);
        externalApi.logEvent("EMAIL_SENT", Map.of("recipient", email));
    }
    
    public void storeItems(List<String> items) {
        if (items == null || items.isEmpty()) {
            throw new IllegalArgumentException("Items cannot be null or empty");
        }
        externalApi.saveBatch(items);
    }
    
    public void refreshCache() {
        externalApi.clearCache();
        externalApi.logEvent("CACHE_CLEARED", Map.of());
    }
}

Why doXxx().when() Instead of when().doXxx()?

// ❌ WRONG - This doesn't work with void methods!

⚠️ Common Pitfalls and Solutions
Pitfall 1: Forgetting doNothing() for Void Methods
java
// ❌ WRONG - Can't use when().thenReturn() for void
when(mockApi.saveUser(anyString(), anyString())).thenReturn(null);
// ERROR! saveUser() is void, cannot return value
// ✅ CORRECT - Use doNothing()
doNothing().when(mockApi).saveUser(anyString(), anyString());
Pitfall 2: Incorrect Exception Type
java
// ❌ WRONG - Exception doesn't match
doThrow(new RuntimeException()).when(mockApi).saveUser(anyString(), anyString());
// Test expecting IllegalArgumentException will fail
// ✅ CORRECT - Throw matching exception
doThrow(new IllegalArgumentException("Invalid user"))
    .when(mockApi).saveUser(anyString(), anyString());
Pitfall 3: Not Specifying Matchers Properly
java
// ❌ WRONG - Mixing exact values and matchers without eq()
doNothing().when(mockApi).saveUser("john", anyString());
// ERROR! Can't mix exact and matchers
// ✅ CORRECT - Use eq() for exact values
doNothing().when(mockApi).saveUser(eq("john"), anyString());
Pitfall 4: Assuming Void Methods Return Something
java
// ❌ WRONG - Trying to capture return value
String result = service.saveData("test");  // ERROR! saveData returns void
// ✅ CORRECT - Void methods don't return anything
service.saveData("test");

🎓 Key Takeaways for Exercise 4
Void methods require special stubbing: Use doXxx().when()
doNothing() is the default behavior (method does nothing)
doThrow() makes void methods throw exceptions
doAnswer() provides custom behavior with argument access
Verification is the same as non-void methods
Don't try to return values from void methods
Use eq() when mixing exact values with matchers
Test exceptions thrown by void methods

Exercise 5: Mocking and Stubbing with Multiple Returns
Scenario:

Why Multiple Returns Are Important?
In real-world applications, methods are often called multiple times with different results:
Pagination: First call returns page 1, second returns page 2
Retry Logic: First call fails, second succeeds
State Changes: First call returns "pending", second returns "completed"
Iteration: Processing multiple items with different responses

package com.example;
import java.util.ArrayList;
import java.util.List;
public class MyService {
    private ExternalApi externalApi;
    
    public MyService(ExternalApi externalApi) {
        this.externalApi = externalApi;
    }
    
    // ============================================
    // METHODS THAT CALL API MULTIPLE TIMES
    // ============================================
    
    // 1. Simple multiple calls
    public String fetchDataMultipleTimes() {
        String result = "";
        for (int i = 0; i < 3; i++) {
            result += externalApi.getData() + " ";
        }
        return result.trim();
    }
    
    // 2. Retry logic
    public String fetchWithRetry(int maxRetries) {
        Exception lastException = null;
        for (int i = 0; i < maxRetries; i++) {
            try {
                return externalApi.getData();
            } catch (Exception e) {
                lastException = e;
                // Log retry attempt
                System.out.println("Retry " + (i + 1) + " failed");
            }
        }
        throw new RuntimeException("All retries failed", lastException);
    }
    
    // 3. Pagination
    public List<User> getAllUsers() {
        List<User> allUsers = new ArrayList<>();
        int page = 0;
        int pageSize = 10;
        
        while (true) {
            List<User> pageUsers = externalApi.getUsers(page, pageSize);
            if (pageUsers == null || pageUsers.isEmpty()) {
                break;
            }
            allUsers.addAll(pageUsers);
            page++;
            if (pageUsers.size() < pageSize) {
                break; // Last page
            }
        }
        return allUsers;
    }
    
    // 4. State transitions
    public String trackOrderStatus(long orderId) {
        Order order = externalApi.getOrderStatus(orderId);
        while (!order.getStatus().equals("COMPLETED") && 
               !order.getStatus().equals("FAILED")) {
            // Wait or do something
            order = externalApi.getOrderStatus(orderId);
        }
        return order.getStatus();
    }
    
    // 5. Conditional processing
    public String processPaymentWithRetry(double amount) {
        try {
            return externalApi.processPayment(amount);
        } catch (Exception e) {
            // Retry with different amount
            return externalApi.processPayment(amount * 0.9);
        }
    }
    
    // 6. Bulk operations
    public List<String> processFileDownloads(List<String> fileNames) {
        List<String> results = new ArrayList<>();
        for (String fileName : fileNames) {
            try {
                byte[] content = externalApi.downloadFile(fileName);
                results.add(fileName + ": Success (" + content.length + " bytes)");
            } catch (Exception e) {
                results.add(fileName + ": Failed - " + e.getMessage());
            }
        }
        return results;
    }
}

How Multiple Returns Work
text
┌─────────────────────────────────────────────────────────────┐
│              MULTIPLE RETURNS MECHANISM                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  when(mock.getData())                                      │
│      .thenReturn("A")                                      │
│      .thenReturn("B")                                      │
│      .thenReturn("C")                                      │
│                                                             │
│  ┌─────────────────────────────────────────────┐          │
│  │  Internal Queue of Responses                │          │
│  │  ┌───────────────────────────────────┐     │          │
│  │  │ Position 1: "A"                   │     │          │
│  │  │ Position 2: "B"                   │     │          │
│  │  │ Position 3: "C"                   │     │          │

⚠️ Common Pitfalls and Solutions
Pitfall 1: Not Enough Stubs
java
// ❌ WRONG - Only one stub but multiple calls
when(mockApi.getData()).thenReturn("A");
mockApi.getData(); // "A"
mockApi.getData(); // "A" (OK, returns last value)
// But if you need different values:
when(mockApi.getData()).thenReturn("A").thenReturn("B");
mockApi.getData(); // "A"
mockApi.getData(); // "B"
Pitfall 2: Overriding Stubs
java
// ❌ WRONG - This overwrites previous stub
when(mockApi.getData()).thenReturn("First");
when(mockApi.getData()).thenReturn("Second"); // Overwrites!
// ✅ CORRECT - Use multiple returns
when(mockApi.getData())
    .thenReturn("First")
    .thenReturn("Second");
Pitfall 3: Complex Return Values
java
// ❌ WRONG - Returning same object reference
List<String> list = Arrays.asList("A", "B");
when(mockApi.getData()).thenReturn(list).thenReturn(list);
// Both calls return the SAME list object!
// ✅ CORRECT - Create new objects each time
when(mockApi.getData())
    .thenReturn(new ArrayList<>(Arrays.asList("A", "B")))
    .thenReturn(new ArrayList<>(Arrays.asList("C", "D")));

Why Order Verification Matters?
When you're testing complex business logic, the order of operations often matters:
Payment Processing: Must validate before charging
Database Operations: Must connect before querying
File Operations: Must open before reading
Email Sending: Must set recipients before sending
State Transitions: Must follow valid state paths

MyService.java (Service with ordered operations)
java
package com.example;
public class MyService {
    private ExternalApi externalApi;
    
    public MyService(ExternalApi externalApi) {
        this.externalApi = externalApi;
    }
    
    // 1. Database workflow
    public void processDatabaseQuery(String query) {
        externalApi.connect();
        try {
            externalApi.executeQuery(query);
        } finally {
            externalApi.disconnect();
        }
    }
    
    // 2. Payment workflow
    public void processPayment(double amount) {
        externalApi.validatePayment(amount);
        externalApi.processPayment(amount);
        externalApi.confirmPayment(12345L);
    }
    
    // 3. File workflow with error handling
    public byte[] readFileSafely(String fileName) {
        externalApi.openFile(fileName);
        try {
            byte[] content = externalApi.readFile(fileName);
            externalApi.closeFile();
            return content;
        } catch (Exception e) {
            externalApi.closeFile();
            throw e;
        }
    }
    
    // 4. Email workflow
    public void sendEmail(String to, String subject, String body) {
        externalApi.createSession();
        externalApi.setRecipient(to);
        externalApi.setSubject(subject);
        externalApi.setBody(body);
        externalApi.send();
        externalApi.closeSession();
    }
    
    // 5. Transaction workflow with rollback
    public void transactionOperation(String query, boolean shouldSucceed) {
        externalApi.beginTransaction();
        try {
            externalApi.executeQuery(query);
            if (!shouldSucceed) {
                throw new RuntimeException("Operation failed");
            }
            externalApi.commit();
        } catch (Exception e) {
            externalApi.rollback();
            throw e;
        }
    }
}

⚠️ Common Pitfalls and Solutions
Pitfall 1: Creating InOrder After Method Calls
java
// ❌ WRONG - InOrder created after calls
mockApi.method1();
mockApi.method2();
InOrder inOrder = inOrder(mockApi);  // Too late! History still recorded though
// ✅ CORRECT - Create InOrder before or after is fine, but be consistent
InOrder inOrder = inOrder(mockApi);
mockApi.method1();
mockApi.method2();
inOrder.verify(mockApi).method1();
inOrder.verify(mockApi).method2();
Pitfall 2: Forgetting to Verify All Calls
java
// ❌ WRONG - Missing verification for some calls
mockApi.method1();
mockApi.method2();
mockApi.method3();
InOrder inOrder = inOrder(mockApi);
inOrder.verify(mockApi).method1();
inOrder.verify(mockApi).method3();  // This will fail! method2 was called in between
// ✅ CORRECT - Verify all calls in order
inOrder.verify(mockApi).method1();
inOrder.verify(mockApi).method2();
inOrder.verify(mockApi).method3();
Pitfall 3: Incorrect Mock Order in InOrder
java
// ❌ WRONG - Mocks in wrong order
mock1.method1();
mock2.method2();
mock1.method3();
InOrder inOrder = inOrder(mock2, mock1);  // Wrong order
inOrder.verify(mock1).method1();  // This will fail because mock2 is expected first
// ✅ CORRECT - Include all mocks in correct order
InOrder inOrder = inOrder(mock1, mock2);
inOrder.verify(mock1).method1();
inOrder.verify(mock2).method2();
inOrder.verify(mock1).method3();

    MyService service = new MyService(api1, api2);
    
    service.processMultiStepWorkflow();
    
    InOrder inOrder = inOrder(api1, api2);
    inOrder.verify(api1).step1();
    inOrder.verify(api2).step2();
    inOrder.verify(api1).step3();
    inOrder.verify(api2).step4();
}
🎓 Key Takeaways
InOrder verifies methods are called in the correct sequence
Create InOrder with inOrder(mock1, mock2, ...)
Verify by calling inOrder.verify(mock).method()
Position tracking ensures strict ordering
Multiple mocks can be tracked together
Combine with times() for repeated operations
Error handling should be verified in order
Stubs count in order verification
Create new InOrder for each test scenario
Clear error messages help debug order issues

Viva Questions
Q1. Why use doThrow() instead of when()?
Because sendNotification() is a void method and does not return a value.
Q2. What does verify() do?
It checks whether a method was called on the mock object.
Q3. What does assertThrows() do?
It verifies that the expected exception is thrown during execution.
